### Category-specific pitfalls — Array (pointer parameter or `a[i]` index access) ###

C1. ACSL does NOT implicitly decay a C array to a pointer in logic terms. Anywhere you'd write `p->arr` (where `arr` is declared `T[N]`) in a pointer context (`\valid(... + range)`, a logic function expecting `T*`, etc.), use `&p->arr[0]`. Same for `\old(p)->arr` → `&\old(p)->arr[0]` and `\at(p,Pre)->arr` → `&\at(p,Pre)->arr[0]`. Frama-C reports `no implicit conversion between a C array and a pointer ... Ignoring logic specification` and the entire spec is silently dropped.

C2. When an `ensures` mentions an array element at the post-state, write `pIp->pkv[i]`; at the pre-state, write `\at(pIp,Pre)->pkv[i]` or `\at(pIp->pkv[i], Pre)`. Mixing them is a frequent source of unprovable goals.

C3. Loop labels: ACSL has `Pre`, `Old`, `Post`, `Here`, `LoopCurrent`. There is NO `LoopEntry`. Use `\at(var, Pre)` instead of `\at(var, LoopEntry)`.

C4. ACSL has NO `\max` / `\min` / `\abs` builtins. When an array postcondition reasons about extrema (max-of-array, abs-bounded), use conditionals (`a > b ? a : b`, `x < 0 ? -x : x`) or define a `logic` function once at the top of the file.
