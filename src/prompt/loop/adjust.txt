### Role: ###

You are an expert in the C language and the Frama-C static analysis tool.

### Task: ###

Your task is to fix an incorrect loop invariant in a given C program based on the provided error messages from Frama-C.

When the "Goal Assertion" is correct, but either "Goal Establishment" or "Goal Preservation" is incorrect, it means the current loop invariant can verify the postcondition but is flawed. You need to adjust the invariant by only modifying the incorrect one.

### Inputs: ###

1.  Specification Error List:

    * Error Description: A concise explanation of the error.

        * "Goal Establishment of Invariant" indicates that Frama-C cannot prove the loop invariant holds in the initial state. This usually means the function's precondition is insufficient to establish the invariant's validity upon entering the loop. You may resolve this either by (a) weakening the loop invariant so that the existing precondition can establish it, or (b) strengthening the function's `requires` precondition so that it implies the invariant on entry — pick whichever better matches the intended behavior.
        * "Goal Preservation of Invariant" indicates that Frama-C cannot prove the loop invariant remains valid after each iteration. This means the invariant is not inductive (it is not preserved by the loop body's operations) or that the invariant becomes false when the loop terminates (the first time the loop condition is false).

    * Error Content: The faulty ACSL annotation.
    

        ```
        {error_str}
        ```
    

2.  C Code with Incorrect ACSL Annotations:

    ```c
    {c_code}
    ```

### Outputs: ###

1.  Error Analysis:

    Provide a detailed analysis of the error and the rationale behind your modification.

2.  Fixed C Code:

    Provide the complete corrected C code with the fixed ACSL annotations, based on the error message and the incorrect annotations in the input C code.

### Rules: ###

1.  Strictly adhere to ACSL syntax: Ensure all corrected annotations comply with the rules of the ACSL specification language.

2.  Do not modify the function body or signature: only change ACSL annotations. You MAY add new top-level `/*@ ... */` blocks before the function — see rule 4. You MAY also adjust the function's `requires` precondition when the Establishment error indicates it is too weak (see Inputs section above).

3.  Do not add any natural language explanations after ACSL annotations

4.  Predicate / logic-function definitions: if the adjusted invariant references a logic function or predicate not already defined in the file (e.g. `list_length(p)`, `sum(a, n)`, `sorted(a)`), you MUST add its complete definition as its own top-level `/*@ logic <type> <name>(...) = ...; */` or `/*@ predicate <name>(...) = ...; */` block, placed BEFORE the function whose invariant uses it. One definition per `/*@ ... */` block. If you skip the definition, Frama-C reports `unbound logic function <name>. Ignoring ... annotation` and drops the entire invariant. Prefer reusing predicates already in the file (e.g. `listrep`, `lseg`) over inventing new helpers.
