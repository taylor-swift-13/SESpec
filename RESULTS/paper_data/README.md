# Paper Raw Data

This directory is the **single canonical root** for every table, figure, and case study in the paper. All SESpec-400 data lives here; the legacy `paper/Results/sespec_400/` location has been removed.

## Layout

```
RESULTS/paper_data/
├── README.md                       (this file)
├── aggregate.csv                   Table I source: 5 methods × 4 models, full_400
│                                   schema: scope, method, model, runs, valid, valid_pct,
│                                           avg_seconds_pass, avg_tokens_pass, avg_api_calls_pass
├── represent_400/                  older / supplementary aggregate (includes LLM-Java rows
│   └── aggregate.csv               and syntax_pass / spec_fail breakdown columns)
│
├── AutoSpec/<model>/<case>/        per-case AutoSpec outputs + summary.json
│   └── per_case.csv                pre-rolled-up per-case verdicts (used by older scripts)
├── SpecGen/<model>/<case>/         per-case SpecGen Java/JML outputs + summary.json
├── LLM-C/<model>/<case>/           Pure-LLM C/ACSL baseline (no SE, no refinement)
├── LLM-Java/<model>/<case>/        Pure-LLM Java/JML baseline (no refinement)
├── SESpec_no_refine/<model>/<case>/   SESpec w/o refinement
├── SESpec/<model>/<case>/          SESpec full pipeline (refine=10)
│
├── judge/                          Pairwise specification strength (4 models)
│   ├── vs_autospec/
│   │   ├── gpt-4o.csv              case_id, verdict, reason
│   │   ├── gpt-5-mini.csv
│   │   ├── gpt-5.4-mini.csv
│   │   ├── gpt-5.csv
│   │   └── summary.csv             per-model aggregate (4 rows)
│   └── vs_specgen/                 (same layout)
│
├── case_studies/                   Case-study source files and outputs
│   ├── case7_search/
│   └── case7_abs_initial/
├── fail_analysis/                  per-case failure-mode reasons
├── c_inputs/, java_inputs/         the 400 C / Java source files
├── backfill_manifest.csv           provenance trail from raw runs to paper_data/
├── slim_manifest.csv               which files survived `slim_paper_data.py`
└── tools/                          scripts that produced subsets of this data
```

`LLM-C` and `LLM-Java` correspond to the "Pure LLM" single-shot baseline rows (no SE injection, no refinement).

## Generation pipeline

Two scripts read from this directory and refresh paper artifacts; both are idempotent:

| Script | Reads | Writes |
|---|---|---|
| `src/scripts/aggregate_all_methods_all_models.py` | `{Method}/{model}/{case}/summary.json` | `aggregate.csv` + main `large_experiment_table.tex` table |
| `src/scripts/append_strength_summary_table.py` | `judge/vs_<baseline>/summary.csv` | appends strength table to `large_experiment_table.tex` |

The main aggregate script also writes only the first table block of `large_experiment_table.tex`; any subsequent `% Auto-generated by …` block (e.g. the strength table) is preserved.

## Mapping from paper artifacts to data paths

| Paper artifact | Location |
|---|---|
| Table I valid counts / cost averages | `aggregate.csv` (and per-case `{Method}/{model}/{case}/summary.json`) |
| Table I "SESpec" generated specs (sample inspection) | `SESpec/<model>/<case>/foo*.c` |
| Table I "AutoSpec" generated specs | `AutoSpec/<model>/<case>/*_merged.c` |
| Table I "SpecGen" generated specs | `SpecGen/<model>/<case>/candidate.java` |
| Strength table per-model verdicts vs AutoSpec | `judge/vs_autospec/<model>.csv` |
| Strength table per-model verdicts vs SpecGen | `judge/vs_specgen/<model>.csv` |
| Strength table aggregate rows | `judge/vs_<baseline>/summary.csv` |
| Case 7 listings (Fig 12–14) | `case_studies/case7_search/{search_source.c, search_sespec_output.c}` |
| Case 7 run log (10 API calls, 76k tokens, 417s) | `case_studies/case7_search/run.log` |
| Sec V RQ1–RQ4 tables (SyGuS, OOPSLA, NLA, numer-S, Frama-C, pIp, list-S) | sibling `RESULTS/SESpec/`, `RESULTS/AutoSpec/` |
| Sec V-D masking experiment (mask plot) | sibling `RESULTS/mask/` |
| Ablation runs | sibling `RESULTS/ablation/` |

The Cases 1–5, Case 6 (MulLoop), Case 8 (Ackermann), Case 9 (Fibonacci) use programs under `src/input/<benchmark>/`; the SESpec annotated outputs are under `SESpec/<model>/<case>/`. The Case 7 source program (`search.c`) is not in any benchmark — it is reproduced verbatim from Preguss~\cite{Wang2025ATO} Fig 3 and was placed under `src/input/preguss_demo/`.

## Schema notes

`{Method}/{model}/{case}/summary.json` (normalized by `aggregate_all_methods_all_models.py:normalize_summary`):
```json
{
  "valid_pass" | "passed" | "result": "pass | ...",    // any of these keys
  "total_seconds" | "seconds": <float>,
  "total_tokens" | "tokens": <int>,
  "call_count" | "api_calls": <int | null>,
  ...                                                   // method-specific extras
}
```

`judge/vs_<baseline>/<model>.csv`:
```
case_id, verdict, reason
```
`verdict ∈ {A_stronger, B_stronger, equal, incomparable, parse_error}`. A = baseline (AutoSpec or SpecGen), B = SESpec. Judge: `gpt-5.4-mini` at temperature 0 with a four-way rubric. Rows for gpt-4o / gpt-5-mini / gpt-5.4-mini are post-human-review counts (paper-validated); gpt-5 rows are LLM-only without human review.

`judge/vs_<baseline>/summary.csv`:
```
model, both_pass_total, B_stronger, A_stronger, equal, incomparable, parse_error
```
Regenerated by `append_strength_summary_table.py` from the per-model CSVs.

## Method-level cost averaging convention

Per-(method, model) cost averages (`avg_seconds_pass`, `avg_tokens_pass`, `avg_api_calls_pass`) are computed over *valid* runs only.

The method-level *cost* aggregates in the Table I cost columns (Time/Tokens/API Calls) are weighted by valid-run count over **the 3 paper-validated models** (gpt-4o, gpt-5-mini, gpt-5.4-mini) only — gpt-5 contributes to per-row Valid counts but is excluded from cost weighting. This matches the published numbers (e.g. AutoSpec Time = 498.4 s) and is enforced by `COST_AVG_MODELS` in `aggregate_all_methods_all_models.py`.
